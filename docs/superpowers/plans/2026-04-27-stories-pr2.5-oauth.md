# Stories Compilation PR 2.5 — OAuth for Third-Party PDS Accounts

> **Status:** in progress 2026-04-27. Builds on PR 1 + PR 2.

**Goal:** Enable automated bookmark fetching for accounts on third-party PDSes, where the simple app-password + service-auth path is rejected by BlueSky's AppView.

**Approach:** Standard AT Protocol OAuth 2.1 + DPoP. Curator does a one-time interactive auth dance in this Claude session to obtain a refresh token, which is stored as a repo Secret. The fetch workflow uses the refresh token to mint short-lived access tokens (DPoP-signed) on each run.

**Non-goals (Phase 1):**
- Refresh-token rotation handling. We'll observe whether BlueSky actually rotates the refresh token; if it does and breaks, we'll add a `BSKY_OAUTH_GITHUB_PAT` to update the Secret from the workflow.
- Multi-account or multi-handle support. One curator, one handle.
- OAuth UI for the homepage. The callback page is just a "paste this back to Claude" landing.

**Spec / CLAUDE.md updates** at the end.

---

## Stages

Each stage is a discrete deliverable; we commit and pause to validate before moving on.

### Stage 1: Static OAuth assets (client metadata + callback page)

**Deliverables:**
- `oauth/client-metadata.json` — public client metadata document. Served by Jekyll passthrough at `https://lightseed.net/oauth/client-metadata.json`.
- `oauth/callback/index.html` — static page that displays the redirect URL with instructions to paste it back to Claude.

**Validation:** after deploy, both URLs must be reachable. The metadata JSON must be valid JSON with the exact `client_id` field equal to its own URL.

### Stage 2: OAuth dependencies

**Deliverables:**
- Update `scripts/requirements.txt` with `pyjwt[crypto]>=2.8`, `cryptography>=42`.

**Validation:** `pytest` passes; nothing else changes yet.

### Stage 3: One-time OAuth init script

**Deliverables:**
- `scripts/oauth_init.py` — interactive helper that:
  - Generates an EC P-256 DPoP keypair (private JWK).
  - Resolves the curator's handle → DID → PDS → authorization server.
  - Constructs the BlueSky authorization URL via PAR (Pushed Authorization Request).
  - Prints the URL for the curator to open.
  - Reads the redirect URL back from stdin.
  - Exchanges the auth code for refresh + access tokens, DPoP-signed.
  - Prints the resulting credentials (refresh token, DPoP private JWK, PDS issuer URL, DID) for the curator to paste into repo Secrets.

**Validation:** Claude (in chat) runs this with the curator's handle, walks through the auth dance, ends with secrets-ready output.

### Stage 4: Update `fetch_saves.py` for OAuth path

**Deliverables:**
- Add an OAuth code path to `fetch_saves.py`:
  - If `BSKY_OAUTH_REFRESH_TOKEN` env var is set, use OAuth (refresh access token, then call AppView with DPoP-signed Authorization).
  - Otherwise, fall back to the existing app-password path (still useful for bsky.social-hosted accounts).
- New env vars: `BSKY_OAUTH_REFRESH_TOKEN`, `BSKY_OAUTH_DPOP_PRIVATE_JWK`, `BSKY_OAUTH_PDS_ISSUER`, `BSKY_OAUTH_DID`.

**Validation:** unit tests for the OAuth refresh flow (mocked HTTP); existing app-password tests still pass.

### Stage 5: Update workflow and walk through end-to-end

**Deliverables:**
- `.github/workflows/fetch.yml` — pass the new env vars from Secrets.
- Curator adds the four new secrets via GitHub UI.
- Manual workflow trigger; ideally returns N>0 records.

**Validation:** real bookmarks committed to inventory.

### Stage 6: Documentation

**Deliverables:**
- Spec section 7 — add OAuth path with errata.
- CLAUDE.md — describe how the curator triggers a re-auth (when refresh token expires, e.g., months from now).
- Plan committed.

## Risks / unknowns

1. **BlueSky's OAuth implementation may have peculiarities** vs. spec (PAR endpoint URL, scope names, DPoP nonce requirements). We'll handle each as it surfaces.
2. **Refresh token expiration / rotation behavior** — observe; pivot if needed.
3. **Scope coverage for bookmarks** — `atproto` + `transition:generic` should grant read access to `app.bsky.bookmark.getBookmarks`. If not, we'll find a more specific scope or pivot.
4. **DPoP nonce** — BlueSky's authorization server may issue a `DPoP-Nonce` header that must be replayed in subsequent DPoP proofs. We'll handle if it shows up.
